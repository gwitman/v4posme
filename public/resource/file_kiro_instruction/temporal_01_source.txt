# INSTRUCCIONES: Módulo de Transferencia de Piezas entre Activos Fijos

## Resumen General
Crear un módulo para transferir, dar de alta y dar de baja piezas (componentes/propiedades) 
entre activos fijos. Seguir el patrón exacto de `app_afx_fixedassent_input`.

## Componente a Utilizar
- Nombre del componente: `tb_transaction_master_transferpiece`
- Este componente debe existir en la tabla `tb_component` para que el sistema lo reconozca.

  ```php
  $objTM["transactionNumber"]   = $this->core_web_counter->goNextNumber($companyID, $branchID, "tb_transaction_master_transferpiece", 0);
  $objTM["transactionID"]       = $this->core_web_transaction->getTransactionID($companyID, "tb_transaction_master_transferpiece", 0);
  ```

  ```php
  $arrayPieceQuantity    = POST "txtPieceQuantity[]";      // cantidad

      $objTMD["transactionID"]          = $objTM["transactionID"];
      $objTMD["transactionMasterID"]    = $transactionMasterID;
      $objTMD["componentID"]            = $objComponent->componentID;
      $objTMD["componentItemID"]        = $value;           // propertyItemID de la pieza
      $objTMD["quantity"]               = $arrayPieceQuantity[$key];
      $objTMD["reference1"]             = $arrayPieceAction[$key];  // "nueva"|"baja"|"transferencia"
      $objTMD["reference2"]             = $arrayPieceName[$key];
      $objTMD["isActive"]               = 1;
      $this->Transaction_Master_Detail_Model->insert_app_posme($objTMD);
  }
  ```

- **LÓGICA DE APLICACIÓN** (cuando el estado es APLICABLE via workflow):
  ```
  Si el workflow valida COMMAND_APLICABLE:
  
  Por cada pieza en el detalle:
      Si accion == "baja":
          → Dar de baja la pieza en tb_component_property (del activo ORIGEN)
          → Component_Property_Model->delete_app_posme($propertyItemID)
          
      Si accion == "nueva":
          → Crear nueva pieza en tb_component_property (en el activo DESTINO)
          → Component_Property_Model->insert_app_posme({
              componentID: componentID del tb_fixed_assent,
              componentItemID: fixedAssentID del activo DESTINO,
              propertyID: catalogDetailID de la pieza,
              name: nombre de la pieza,
              value: cantidad,
              type: tipo,
              isActive: 1
            })
            
      Si accion == "transferencia":
          → Dar de baja la pieza del activo ORIGEN
          → Component_Property_Model->delete_app_posme($propertyItemID_origen)
          → Crear la pieza en el activo DESTINO
          → Component_Property_Model->insert_app_posme({...datos de la pieza en destino...})
  ```

## 5. LÓGICA DE NEGOCIO (Reglas de Aplicación)

Cuando el estado del documento pasa a APLICABLE (validado por workflow):

### Acción "baja"
- Se busca la pieza en `tb_component_property` donde:
  - `componentItemID` = fixedAssentID del activo ORIGEN (`sourceWarehouseID`)
  - `propertyItemID` = valor almacenado en el detalle (`componentItemID`)
- Se ejecuta: `Component_Property_Model->delete_app_posme($propertyItemID)` (soft delete, isActive=0)

### Acción "nueva"  
- Se crea un nuevo registro en `tb_component_property`:
  - `componentID` = componentID de "tb_fixed_assent"
  - `componentItemID` = fixedAssentID del activo DESTINO (`targetWarehouseID`)
  - `propertyID` = ID del catálogo de la pieza
  - `name` = nombre de la pieza
  - `value` = cantidad
  - `isActive` = 1

### Acción "transferencia"
- Combinación de baja + nueva:
  1. Dar de baja la pieza del activo ORIGEN (delete_app_posme del propertyItemID)
  2. Crear la pieza en el activo DESTINO (insert_app_posme con datos nuevos)

## 6. VALIDACIONES

### En el formulario (JavaScript):
- Fecha es obligatoria
- Estado es obligatorio
- Al menos 1 pieza en el detalle
- Cada pieza debe tener seleccionada una acción (nueva/baja/transferencia)
- Si acción es "baja" o "transferencia" → activo origen debe estar seleccionado
- Si acción es "nueva" o "transferencia" → activo destino debe estar seleccionado

### En el servidor (PHP):
- Validación de autenticación y permisos
- Validación de workflow (editable, eliminable, aplicable)
- Validar que el activo origen/destino exista
- Validar que la pieza exista en el activo origen antes de darla de baja
- Transacciones con `$db->transStart()` y `$db->transCommit()` / `$db->transRollback()`

## 8. NOTAS IMPORTANTES

- Seguir exactamente el patrón de `app_afx_fixedassent_input` para la estructura del controlador
- Los campos `sourceWarehouseID` y `targetWarehouseID` de `tb_transaction_master` se reutilizan para almacenar el activo origen y destino respectivamente
- El activo origen y destino son NO obligatorios (depende de la acción seleccionada en las piezas)
- Se pueden agregar múltiples piezas en una sola transacción
- La aplicación de altas/bajas solo se ejecuta cuando el workflow permite (estado APLICABLE)
- Usar `$this->core_web_auditoria->setAuditCreated()` al insertar
- Usar transacciones de base de datos para garantizar integridad
